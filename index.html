01 · Brief 解读台
把模糊需求变成清晰的作战任务书，识别隐性诉求，生成客户 QA 追问清单
目的 & 价值
消除 Brief 理解偏差，避免方向跑偏。客户写的 Brief 通常只有表面目标，AI 负责挖出没说出来的隐性诉求，减少中途大改的返工成本。
用户操作步骤
Step 1 上传 Brief 文件（PDF / PPT / Word）或粘贴飞书链接
Step 2 AI 自动识别方案类型，显示置信度（如：Social 85%）
Step 3 输出结构化任务卡：项目背景 · 核心目标 · 受众 · KPI · 时间线 · 预算区间
Step 4 「透过现象」区块：AI 标注隐性诉求和潜在风险点（红色高亮）
Step 5 QA 追问清单：5-8 个需向客户确认的问题，可一键复制发给客户
Step 6 人工确认 / 修改后，点击「进入框架生成」带入 Step 02
AI 处理链路
① 文档解析：按文件类型调用对应 Skill，提取纯文本
② 类型分类：Brief 特征 → 方案类型（含组合识别，如 Social+KOL）
③ 结构提炼：6维度结构化输出（含「未明确」标注）
④ 隐性挖掘：对照历史中标经验，识别客户真实诉求 vs 表面诉求
⑤ QA 生成：按 Brief 类型（新车上市 / 年度品牌 / 活动营销）调用对应追问模板
技术依赖清单
类型 工具 / 资源 用途
Skill pdf Skill 解析 PDF 格式 Brief
Skill pptx Skill 解析 PPT 格式 Brief
Skill docx Skill 解析 Word 格式 Brief
MCP lark-doc Skill 飞书链接直读，免下载
Prompt Brief 分类模板 ×4 新车上市 / 年度品牌 / 活动营销 / 危机公关
Prompt QA 追问模板 按 Brief 类型调用对应追问逻辑
知识库 历史 Brief → 中标关系 隐性诉求识别的参照底座
✅ 无需付费外部 API，全部依赖本地 Skill + Prompt。飞书 Brief 需要飞书 Token 授权（已有 lark-doc Skill 覆盖）。

02 · 框架生成台
按方案类型调 Skill，一键生成中标框架骨架
目的 & 价值
30份中标方案提炼的四类框架直接可用，骨架搭建从 1 天压到 10 分钟，新人也能用专业结构起步。
用户操作步骤
Step 1 自动带入任务卡，显示识别的方案类型
Step 2 确认或手动调整（可选组合类型，如 PR+Social）
Step 3 生成完整框架骨架，每个模块展示「此处要论证什么」
Step 4 每个模块旁显示「推荐金块」气泡（来自金块库的相关素材）
Step 5 可拖拽调整模块顺序，增删章节
Step 6 确认骨架，带入洞察引擎 + 策略工作台
技术依赖
Skill auto-strategy Skill — 四类框架调用（Social六段论 / PR十步链 / Media五段式 / KOL六步链）
Skill strategy-workflow Skill — 6步链路辅助
API 向量数据库（Chroma 本地 / Pinecone）— 金块语义检索，框架模块 → 相关金块召回
知识库 框架 × 金块映射表 — 哪类金块适合填充哪个框架模块位置

03 · 洞察引擎
输入数据需求 → 后台路由至对应平台 → 自动分析返回洞察
目的 & 价值
洞察是方案有没有说服力的核心。过去策略人要自己分别登录千瓜、飞瓜、汽车之家查数据，拼在一起，耗时 1-2 天。这里变成：说一句话，系统自动判断去哪里拿数据、拿回来分析。
交互方式（用户视角）
💬 用户输入："帮我分析理想L9近3个月在小红书的KOL投放情况，以及竞品问界M9的对比"
↓
🤖 系统自动：识别需求 → 路由至千瓜API → 拉取数据 → AI分析 → 输出洞察卡片
AI 处理链路
① 意图解析：识别数据类型（KOL/内容/口碑/行业/竞品）+ 平台 + 时间范围 + 品牌
② 数据路由：自动匹配对应 API，组合多源时并行调用
③ 数据分析：结构化数据 → 趋势判断 + 异常识别 + 竞品差距
④ 洞察提炼：数字背后的「为什么」和「意味着什么」
⑤ 基准召回：对照金块库历史数据基准，给出参考值

04 · 策略工作台
骨架 + 洞察 → 方案主线定调 → 金块召回 → 完整策略叙事
目的 & 价值
整个产品最核心的生产力模块。把骨架和洞察组装成有逻辑、有说服力的策略内容，同时把 30 份方案沉淀的金块作为弹药按需召回，让输出有历史依据而不是凭空生成。
用户操作步骤
Step 1 主线定调：AI 基于洞察生成 3 个策略方向（每个含大创意 + 核心主张 + 叙事逻辑），人工选一个
Step 2 逐模块生成：按骨架顺序，每个模块旁自动展示匹配的 S/A 金块卡片
Step 3 金块操作：点击金块可选择「直接引用」或「让 AI 改写融入」
Step 4 内容编辑：富文本区，可直接修改 AI 输出，也可输入指令让 AI 重写
Step 5 方案自检：完成后一键自检，对照框架规则标出逻辑缺口（红色高亮）
Step 6 策略文档确认后流转至「执行 & 创意」和「输出闭环」
AI 处理链路
① 主线生成：洞察 × 骨架类型 × 品牌 DNA → 策略方向候选
② 语义召回：当前模块文本 → 向量数据库检索 top-5 相关金块
③ 内容填充：调用汽车行业 System Prompt，生成有行业质感的策略文案
④ 自检逻辑：逐项对照框架规则库，输出「完整 / 缺失 / 薄弱」三色标注
技术依赖清单
类型 工具 / 资源 用途
Skill auto-strategy Skill 各类型策略模块内容生成
Skill strategy-workflow Skill 6步链路（主线定调环节）
API 向量数据库（Chroma） 金块语义召回，按模块上下文检索
Prompt 汽车行业 System Prompt 话语体系：让输出有行业质感，不像通用 AI
Prompt 方案自检规则库 按框架类型的完整性 & 逻辑规则
知识库 客户 DNA 库 各客户品牌调性、历史主张，防止风格跑偏
⚠️ 策略主导的核心节点：主线定调（Step 1）AI 给选项，人工必须拍板。这一步不能全自动，方向选错后面全错。

05 · 执行 & 创意
策略落地规划 + 创意发散 + 视觉出图
目的 & 价值
消除策略和执行之间的断层。直接输出可操作的传播节点规划和有视觉感的创意方向，让甲方看到「怎么做」而不只是「想什么」。
用户操作步骤
Step 1 执行 Roadmap：选择执行周期（1/3/6个月），AI 生成带传播节点的时间线，可拖拽调整
Step 2 预算结构：AI 按方案类型推荐各渠道比例（参照历史数据金块），可手动调整总预算
Step 3 创意方向：生成 5-8 个创意角度，每个含一句话概念 + 内容形式建议
Step 4 视觉出图：选一个创意方向 → 输入视觉风格描述 → 调用 GPT image 生成参考图
技术依赖
API OpenAI Image API（DALL-E 3 / GPT-4o image）— 创意视觉参考图生成，付费调用
Prompt 排期生成 Prompt — 按节点类型（上市 / 大促 / 节日）生成传播时间线
知识库 数据基准金块（61个） — 历史预算比例、CPM 参考值、渠道效率数据
Prompt 创意发散 Prompt — 结合策略主线生成差异化创意方向，非通用发散

06 · 输出闭环
策略内容 → 飞书文档协作审阅 → 框架确认 → PPT 生成 + 配图 → 提案话术，全链路一键交付
飞书文档 策略大纲协作
Step 1 点击「同步飞书」，自动在飞书创建结构化文档
Step 2 策略各章节分块写入，保留标题层级
Step 3 自动 @ 相关审阅人（AE / CD / 客户），发飞书通知
Step 4 团队在飞书内批注修改，确认框架
MCP lark-doc Skill — 飞书开放平台 Docs API，创建文档 + 写入内容
MCP lark-im Skill — 飞书 IM API，自动发送通知 + @审阅人
MCP lark-contact Skill — 获取团队成员信息用于 @
PPT 框架确认后出稿
Step 1 飞书框架确认后，点击「生成 PPT」
Step 2 选择客户品牌模板（品牌色 / 字体 / 版式）
Step 3 AI 逐页生成：页面主题 + 核心信息点 + 图表建议
Step 4 每页视觉位自动触发 GPT image 生成配图候选
Step 5 人工选图确认 → 合并为完整 .pptx 文件下载
Skill pptx Skill — .pptx 文件生成，支持模板套用 + 内容填充
MCP lark-slides Skill — 飞书演示文稿（备选，适合直接飞书演示场景）
API OpenAI Image API — 每页配图生成（付费，按页数计）
资源 PPT 品牌模板库 — 各客户模板 .pptx，需人工维护
话术 提案口播稿
Step 1 选择提案时长（5 分钟 / 15 分钟 / 30 分钟）
Step 2 选择受众类型（客户市场部 / CMO / 外部评审）
Step 3 AI 生成逐页口播稿，含「重点强调」和「预判追问」
Step 4 可针对某页重新生成，或指令修改语气（更口语 / 更正式）
Step 5 导出 Word 格式，附 PPT 页码对应
Prompt 话术生成 Prompt — 按时长 × 受众类型 × 方案类型的话术模板
Skill docx Skill — 话术导出为 Word 文档
知识库 客户偏好记录 — 该客户过去对哪类表达有正面反馈
💡 闭环逻辑：飞书文档是「人工确认节点」，不是装饰步骤——框架没有在飞书上确认，PPT 就不生成。这个设计避免了「AI 生成了一半内容发现方向跑偏」的浪费。话术在 PPT 确认后生成，保证口播和页面完全对应。
